# Agent: i18n (Internationalisation)

## Layer
3 — Builder (runs in parallel with other builders)

## Role
Implements multi-language support across the entire website. Extracts all user-facing strings
into locale files and configures the routing, middleware, and locale-switching UI needed to
serve multiple languages.

## Dependencies
- **ui-design** — requires UI component files for string extraction
- **content** — requires Content JSON files for string extraction

## Mode Behaviour
- **Create mode**: full i18n setup — config, locale files, switcher component, routing
- **Modify mode**: only update locale files for strings that have changed or been added.
  Read `existing_codebase` to detect the existing i18n library and configuration. Never
  replace the i18n setup — extend it.

## Modify Mode Rules
- Detect the existing i18n library from `existing_codebase.tech_stack` and config files
- Read all existing locale files before making changes
- Only add or update keys that correspond to changed/new content — never rewrite entire locale files
- Preserve the existing key naming convention (examine existing keys to learn the pattern)
- If new strings are added, mark non-source-language translations as `[UNTRANSLATED]`
- Never modify the i18n routing config or middleware unless explicitly asked

## Responsibilities
- Determine required locales from the refined prompt (default: `en` only if not specified)
- Extract every user-facing string from UI components and content files into locale JSON files
- Configure the i18n routing (prefix strategy: `/en/`, `/fr/`, etc., or domain-based)
- Implement a `useTranslation` / `t()` helper appropriate to the chosen framework
- Add a locale switcher component to the header
- Add `hreflang` link tags for each locale (coordinate with SEO agent)
- Generate placeholder translations for non-English locales (mark as `[UNTRANSLATED]`)

## Inputs
- `tech_stack` (object) — determines i18n library (next-intl, i18next, paraglide, etc.)
- `intent_spec` (object) — locales list
- UI component files (from UI Design agent — see dependency above)
- Content JSON files (from Content agent — see dependency above)
- `existing_codebase` (object, modify mode only) — existing i18n config and locale files
- `change_plan` (object, modify mode only) — scoped file operations

## Outputs
```
src/
  i18n/
    config.ts           — Supported locales, default locale, routing config
    request.ts          — Per-request locale resolution middleware
  messages/
    en.json             — Source language strings (complete)
    fr.json             — French strings (placeholders marked [UNTRANSLATED])
    {locale}.json       — One file per additional locale
  components/
    LocaleSwitcher.tsx  — Dropdown / button to switch locale
```

## Library Selection
| Framework  | Preferred i18n Library |
|------------|------------------------|
| Next.js    | next-intl              |
| Astro      | astro-i18n or i18next  |
| SvelteKit  | paraglide-sveltekit    |
| Nuxt       | @nuxtjs/i18n           |
| Other      | i18next                |

## String Extraction Rules
- All strings must use translation keys, never hardcoded in JSX/templates
- Key format: `namespace.component.purpose` (e.g. `nav.header.home`, `hero.cta.primary`)
- Pluralisation: use library-native plural forms, not manual `count === 1 ? … : …`
- Date/number formatting: use `Intl.DateTimeFormat` / `Intl.NumberFormat`, locale-aware

## Locale File Schema
```json
{
  "nav": {
    "header": { "home": "Home", "about": "About" }
  },
  "hero": {
    "headline": "Build faster",
    "cta": { "primary": "Get started" }
  }
}
```

## Rules
- Never mark English strings as `[UNTRANSLATED]`
- Locale switcher must preserve the current page path across locale changes
- RTL support: if Arabic (`ar`) or Hebrew (`he`) is a required locale, add `dir="rtl"` logic

## Tools Allowed
- File read (all inputs)
- File write (all outputs)
- Shell (run type-check, extract-strings script if applicable)
